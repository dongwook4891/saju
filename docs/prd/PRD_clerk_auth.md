# PRD: Clerk 인증 시스템

**관련 문서**: [FLOW_clerk_auth.md](../flow/FLOW_clerk_auth.md) | [SPEC_clerk_auth.md](../spec/SPEC_clerk_auth.md)

## 목적
사주 서비스의 회원 전용 기능(결제, 결과 저장, 구독 관리 등)을 안전하게 제공하기 위한 인증 시스템을 만든다.

## 해결하려는 문제
- 비회원도 사주 조회는 가능하지만, 결제/저장/재조회는 인증이 필요하다.
- 로그인 과정이 복잡하면 결제 직전 이탈이 커진다.
- 로그인은 되었더라도, 탈퇴/차단 같은 서비스 상태까지 함께 관리해야 한다.

## 제품 목표
1. 사용자는 Google 또는 Email로 쉽게 가입하고 로그인할 수 있어야 한다.
2. 비회원이 결제를 시도하면 자연스럽게 로그인으로 이어지고, 로그인 후 원래 화면으로 돌아와야 한다.
3. 재방문 시 로그인 상태가 유지되어야 한다.
4. 우리 서비스는 로그인 성공 여부와 별개로, 사용자 상태(`active`, `withdrawn`, `blocked`)를 관리할 수 있어야 한다.
5. Clerk와 Supabase 사용자 정보가 어긋나지 않도록 안정적으로 동기화되어야 한다.

## 주요 사용자 시나리오

### 1. 신규 가입
- 사용자가 회원가입 버튼을 누른다.
- Google 또는 Email로 가입한다.
- 가입이 끝나면 바로 서비스에 들어간다.

### 2. 기존 사용자 로그인
- 사용자가 로그인 버튼을 누른다.
- 기존에 쓰던 방법으로 로그인한다.
- 로그인 후 원래 보려던 화면으로 돌아간다.

### 3. 결제 전 인증
- 비회원이 결제를 누른다.
- 로그인 화면으로 이동한다.
- 로그인 후 결제 화면으로 돌아와서 이어서 진행한다.

## 핵심 정책

### 1. 로그인 수단
- 1차 범위에서는 Google, Email(비밀번호 로그인)만 제공한다.
- Kakao, Facebook은 이번 범위에서 제외한다.

### 2. 계정 상태 정책
- `active`: 정상 사용 가능
- `withdrawn`: 탈퇴 상태, 재가입 가능
- `blocked`: 서비스 이용 불가

### 3. 이메일 중복 정책
- 같은 이메일은 하나의 서비스 계정만 허용한다.
- 이미 같은 이메일의 계정이 있으면, 새로 여러 개 만드는 대신 기존 방식으로 로그인하도록 안내한다.
- 단, 기존 계정이 `withdrawn`이면 재가입을 허용한다.

중요:
- 이것은 제품 정책이다.
- 실제 구현에서는 Clerk 가입이 먼저 완료될 수 있으므로, 서비스 진입 단계에서 최종적으로 한 번 더 확인해 막을 수 있어야 한다.

쉽게 말하면:
- "같은 이름표를 가진 회원카드는 한 장만 둔다"가 정책이다.
- 하지만 문 앞 기계(Clerk)가 먼저 카드를 만들어줄 수 있으니, 안쪽 데스크(Supabase)에서 마지막 확인을 한 번 더 해야 한다.

### 4. 차단 계정 정책
- 차단 사용자는 로그인 자체가 될 수 있어도, 우리 서비스 이용은 막아야 한다.
- 차단 시 사용자에게 고객센터 안내를 보여준다.

## 주요 기능
1. 로그인/회원가입
2. 보호 페이지 접근 제어
3. 결제 전 인증 유도
4. 사용자 상태 기반 접근 제어
5. Clerk ↔ Supabase 사용자 정보 동기화

## 엣지케이스 대응

### 1. 이메일 중복
- 목표:
  - 사용자가 같은 이메일로 여러 서비스 계정을 만들지 않게 한다.
  - 가능하면 기존에 쓰던 로그인 방법으로 자연스럽게 유도한다.
- 처리 방향:
  - 사전 확인이 가능한 화면에서는 먼저 안내한다.
  - 이미 Clerk 가입이 된 경우에도, 서비스 진입 시점에 최종 확인하여 중복 계정 사용을 막는다.

### 2. Clerk 성공, Supabase 동기화 실패
- 사용자는 로그인 또는 가입에 성공했지만, 우리 쪽 회원 명단 저장이 실패할 수 있다.
- 이 경우 자동 재시도(최대 3회)가 필요하다.
- 계속 실패하면 사용자에게 일시적 오류를 안내하고 다시 시도할 수 있게 한다.

### 3. 네트워크/OAuth 오류
- 로그인 실패 이유를 사용자가 이해할 수 있게 안내한다.
- 다시 시도할 수 있는 흐름을 제공한다.

### 4. 탈퇴/차단 계정
- `withdrawn`: 재가입 또는 복구 가능한 흐름 제공
- `blocked`: 이용 제한 안내 및 고객센터 연결

### 5. 필수 약관 미동의
- 필수 약관 미동의 시 추가 안내 또는 동의 단계로 이동한다.

## 사용자 경험 원칙
- 로그인은 최대한 짧고 간단해야 한다.
- 로그인 후에는 가능한 한 원래 보던 화면으로 되돌아가야 한다.
- 오류가 생겨도 사용자는 "지금 뭘 해야 하는지" 바로 이해할 수 있어야 한다.
- 서비스 정책으로 막히는 경우에도 이유와 다음 행동을 분명히 보여줘야 한다.

## 성공 지표
- 로그인 완료율 95% 이상
- 결제 직전 이탈률 5% 이하
- 인증 관련 고객 문의 주 5건 이하
